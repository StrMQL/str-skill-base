# Skill Base 小红书轮播卡片设计稿

## 背景

本设计稿用于生成一个独立的静态 HTML 页面，服务于 Skill Base 的小红书内容运营。页面核心目标不是充当官网，也不是做在线编辑器，而是稳定地产出一组可复用、可单独导出的宣传卡片。

第一期主题聚焦“为什么团队需要 Skill Base”，面向混合受众：既要让技术负责人和工程团队认可问题判断，也要让产品、QA、业务同学一眼看懂这是在解决什么管理问题。

## 目标

本页面只承担三项职责：

1. 展示一组适合小红书轮播发布的竖版卡片
2. 支持将每一张卡片单独导出为 PNG 图片
3. 使用现有系统截图作为产品存在的证据素材

## 非目标

以下内容明确不做：

1. 不做多语言切换
2. 不做后台管理或在线编辑
3. 不做复杂模板配置系统
4. 不做长图优先导出，单卡导出是主路径
5. 不为未来“也许会有”的场景增加抽象层

## 设计原则

### 1. 一张卡只讲一件事

每张卡只保留一个核心判断，避免信息密度失控。标题负责抓注意力，正文负责解释，截图负责证明，不允许一张卡同时承担两个主叙事。

### 2. 非技术用户优先可读

文案不依赖工程术语才能成立。即使用户不理解 Skill、CLI、版本号等词，也能通过上下文读懂产品价值。

### 3. 截图是证据，不是主角

截图只出现在证明卡或少量说明卡中，用来支撑“这是真产品，不是概念图”。页面不以大面积截图拼贴为核心视觉。

### 4. 版式固定，内容可替换

后续第二期、第三期沿用同一套卡片骨架，只替换卡片数据，不重做结构和导出逻辑。

## 信息结构

### 页面层级

页面结构保持简单：

1. 顶部工具条
2. 卡片列表容器
3. 每张卡右上角单独导出按钮
4. 页面级“导出全部”按钮

### 卡片数据结构

页面以一组内嵌卡片数据驱动渲染，不额外拆 JSON 文件。

```js
const cards = [
  {
    id: 'cover',
    type: 'cover',
    eyebrow: 'Skill Base / 01',
    title: '为什么团队需要 Skill Base',
    desc: '...',
    points: ['...', '...', '...'],
    screenshot: 'docs/images/skill-base-home.png',
    screenshotAlt: 'Skill Base 首页',
    exportName: 'skill-base-cover'
  }
];
```

### 字段约束

1. `id` 用于 DOM 标识与导出命名
2. `type` 只允许 `cover`、`pain`、`solution`、`proof`、`cta`
3. `points` 最多 3 条，超过即视为文案失焦
4. `screenshot` 仅在需要时填写，为空则不渲染截图区
5. `exportName` 用于下载文件名，避免后续手动改名

## 第一期开卡规划

第一期固定为 7 张卡，顺序如下：

### 1. 封面卡 `cover`

- 主题：为什么团队需要 Skill Base
- 作用：用最短的话引出问题，建立受众入口
- 内容重点：团队明明已经有经验，但 AI 并不会自动理解这些经验

### 2. 痛点卡 `pain-1`

- 主题：AI 不懂公司的业务黑话、流程和规范
- 作用：让用户意识到“模型聪明”不等于“懂你们公司”

### 3. 痛点卡 `pain-2`

- 主题：知识散在 Git、文档、聊天记录和个人经验里
- 作用：解释团队知识无法稳定复用的原因

### 4. 痛点卡 `pain-3`

- 主题：换项目、换 IDE、带新人，全靠复制粘贴
- 作用：把问题落到具体协作成本

### 5. 解决卡 `solution`

- 主题：Skill Base 把团队经验变成可发布、可安装、可更新的 Skill
- 作用：明确产品解法，不讲空话

### 6. 证据卡 `proof`

- 主题：这不是概念，已经有可用界面和版本管理
- 作用：展示真实系统截图，提高可信度
- 截图来源：`docs/images/skill-base-home.png`、`docs/images/skill-detail.png`

### 7. 收尾卡 `cta`

- 主题：先让 AI 学会你们公司怎么做事
- 作用：用一句可传播的话收尾，适合社媒传播

## 第二期开卡规划

第二期主题聚焦“Skill Base 的核心优势”。这一期不再重复铺陈“为什么会乱”，而是直接回答“它到底比现有做法好在哪”。叙事重心从问题共鸣切换到能力判断，但仍然保持非技术用户可读。

第二期固定为 7 张卡，顺序如下：

### 1. 封面卡 `cover`

- 主题：Skill Base 不只是存文件，它在统一团队 AI 工作方式
- 作用：给第二期建立更明确的价值视角
- 内容重点：强调这不是普通网盘，不是另一个文档站

### 2. 优势卡 `benefit-1`

- 主题：私有化部署，团队经验不必上传公共平台
- 作用：回应“为什么不用公开 Skill 市场或公共仓库”
- 内容重点：内部规范、业务流程、私有术语都能放在自己的网络环境里

### 3. 优势卡 `benefit-2`

- 主题：一次发布，多处安装，不再手动同步
- 作用：突出分发效率
- 内容重点：Skill 发布一次后，多个项目和多个 IDE 都能更新到同一套规则

### 4. 优势卡 `benefit-3`

- 主题：支持多 IDE，不把团队绑死在一个工具上
- 作用：强调跨工具协作价值
- 内容重点：Cursor、Claude Code、Qoder、Windsurf、OpenCode 等可以共用团队资产

### 5. 优势卡 `benefit-4`

- 主题：CLI 和 Web 同时可用，研发和非研发都能参与
- 作用：扩大适用人群，不把产品定义成“只有工程师能用”的工具
- 内容重点：研发走命令行，产品和 QA 走 Web，各用各的入口

### 6. 证据卡 `proof`

- 主题：真实界面 + 真实版本记录，证明这套机制能落地
- 作用：用界面截图和版本概念支撑“可管理、可追踪”这件事
- 截图来源：优先使用 `docs/images/skill-base-home.png`、`docs/images/skill-detail.png`

### 7. 收尾卡 `cta`

- 主题：把团队标准变成可安装资产，而不是群消息附件
- 作用：输出一个更偏“资产化”视角的传播句子

## 第三期开卡规划

第三期主题聚焦“Skill Base 怎么用，上手有多简单”。这期不再解释抽象价值，而是把产品拉回操作层，让用户感受到门槛低、路径清楚、当天就能跑起来。

第三期固定为 7 张卡，顺序如下：

### 1. 封面卡 `cover`

- 主题：Skill Base 怎么用，其实比你想得简单
- 作用：降低心理门槛
- 内容重点：告诉用户这不是一个需要复杂部署和培训的新系统

### 2. 流程卡 `flow-1`

- 主题：第一步，启动服务
- 作用：传达“先跑起来”比“先懂全部概念”更重要
- 内容重点：一行 `npx skill-base -d ./skill-data -p 8000` 就能启动本地或内网服务

### 3. 流程卡 `flow-2`

- 主题：第二步，配置 CLI 并登录
- 作用：补齐个人使用入口
- 内容重点：`skb init`、`skb login` 组成最小可用路径

### 4. 流程卡 `flow-3`

- 主题：第三步，像 npm 一样搜索、安装、更新
- 作用：借用熟悉心智，降低学习成本
- 内容重点：`skb search`、`skb install`、`skb update` 是核心日常动作

### 5. 流程卡 `flow-4`

- 主题：第四步，把团队经验发布成 Skill
- 作用：把“使用者”自然过渡到“贡献者”
- 内容重点：只要目录里有 `SKILL.md`，就能通过 `skb publish` 发布版本

### 6. 证据卡 `proof`

- 主题：CLI 和 Web 都能用，流程不是纸上谈兵
- 作用：用截图和命令片段说明这套操作链条真的很短
- 截图来源：优先使用 `docs/images/skill-base-home.png`，必要时配合 README 中的命令示例文案

### 7. 收尾卡 `cta`

- 主题：今天把服务跑起来，明天就能让团队 AI 说人话
- 作用：给出一个直接、带行动感的结尾

## 视觉规范落地

本页面严格遵循用户提供的视觉规范，并落为可实现的静态页面规则。

### 字体

统一使用：

```css
font-family: 'Inter', 'Source Han Sans SC', 'Alibaba PuHuiTi', sans-serif;
```

说明：

1. 英文和数字优先使用 Inter
2. 中文优先使用思源黑体，阿里普惠体作为兜底
3. 不引入非开源商用字体

### 颜色

核心色值固定如下：

1. 品牌蓝：`#1D4ED8`
2. 点缀黄：`#FDE047`
3. 主文本：`#111827`
4. 次文本：`#9CA3AF`
5. 页面背景：`#F8FAFC`
6. 卡片背景：`#FFFFFF`
7. 边框浅灰：`#F3F4F6`
8. 分割虚线：`#E5E7EB`

### 卡片尺寸

单卡按小红书轮播习惯使用竖版比例，目标画布尺寸为 `1242 x 1660`。HTML 内部按同等比例实现，导出时保持清晰度。

### 卡片骨架

每张卡片统一为三段布局：

1. 顶部：eyebrow、小标签或阶段编号
2. 中部：主标题、说明文本、强调装饰线
3. 底部：观点列表或截图证据区

### 共用视觉元素

1. 左侧蓝色强调线：宽 `4px`，圆角 `2px`
2. 卡片圆角：`12px`
3. 卡片阴影：`0 20px 40px -10px rgba(0,0,0,0.05)`
4. 截图区套壳：Mac 风格窗口头，包含红黄绿三点
5. 标题区与内容区使用浅灰色虚线做节奏分隔

### 截图使用规则

1. 截图只放在 `proof` 卡与少量需要证据的卡
2. 截图统一包裹在 mockup window 内
3. 不叠加夸张滤镜，不做过度裁切
4. 多张截图时优先保持统一宽度和统一边框

## 交互设计

### 页面级操作

页面顶部只提供一个高频操作：

1. 导出全部卡片

### 卡片级操作

每张卡片右上角提供“下载图片”按钮，点击后只导出当前卡片，不导出整个页面。

### 导出实现

导出逻辑采用 `html2canvas`：

1. 选中目标卡片 DOM 节点
2. 生成高分辨率 canvas
3. 转为 PNG
4. 通过 `exportName` 自动下载

### 导出约束

1. 导出单位以单卡为主
2. “导出全部”按卡片顺序逐张下载
3. 失败时给出轻量提示，不阻塞页面其他操作

## 内容策略

### 文案风格

文案必须满足以下约束：

1. 不堆砌术语
2. 不写“赋能、闭环、生态协同”这类空话
3. 标题短，判断明确
4. 正文使用直接陈述句，避免营销腔过重

### 叙事路线

第一期统一采用：

1. 提出问题
2. 拆出问题为什么长期存在
3. 给出直接解法
4. 用真实界面收尾证明
5. 留下一句可传播的结论

## 组件划分

为控制复杂度，页面只拆分为以下逻辑单元：

### 1. 页面壳层

负责顶部标题、操作按钮和卡片容器布局。

### 2. 卡片渲染器

根据 `cards[]` 逐张渲染卡片。内部只根据 `type` 做少量样式差异，不做复杂嵌套模板系统。

### 3. 截图渲染块

负责渲染 mockup window 和截图图片。无截图时不渲染该区域。

### 4. 导出模块

负责调用 `html2canvas` 并触发下载。导出逻辑和卡片渲染分离，避免后续维护时互相污染。

## 错误处理

错误处理保持克制：

1. 截图资源加载失败时，卡片显示占位说明，不让页面崩掉
2. 导出失败时，显示简单错误提示，例如“导出失败，请重试”
3. 单张卡导出失败不影响其他卡继续导出

不引入复杂的全局错误状态管理。这个页面不配那种架构。

## 验证方案

完成实现后需要验证以下内容：

1. 页面在本地浏览器正常打开
2. 所有 7 张卡片正常渲染
3. `docs/images/skill-base-home.png` 与 `docs/images/skill-detail.png` 正常显示
4. 单张卡“下载图片”功能可用
5. “导出全部”可按顺序导出全部卡片
6. 导出图片中的字体、边距和截图不出现错位

## 后续扩展方式

第二期、第三期只允许通过替换 `cards[]` 内容来扩展：

1. 第二期可聚焦“Skill Base 的核心优势”
2. 第三期可聚焦“Skill Base 怎么用，上手有多简单”

扩展要求：

1. 不新增第二套版式
2. 不新增第二套导出逻辑
3. 不提前抽象出模板编辑器
4. 第二期和第三期仍然保持 7 张卡的节奏，不为了凑信息量强行加页
5. 三期内容的封面卡、证据卡、收尾卡位置保持一致，降低维护成本
6. 三期的变化主要发生在中间 4 张内容卡，保证结构稳定、文案可替换

保持同一数据结构和导出机制，才是长期可维护的做法。
